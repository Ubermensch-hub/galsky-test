# Прошивка телематического устройства - тестовое задание

Каркас прошивки трекера. 10 модулей работают одновременно на кооперативном
планировщике и обмениваются данными через статические каналы. Бюджет ОЗУ -
128 КБ - контролируется на этапе компиляции. C++17, без кучи, без потоков ОС.
Проект собирается и тестируется на desktop, платформенные различия
разрешаются только на уровне линковки, без `#ifdef` в переносимом коде.

Специфические термины и сокращения (SPSC, ISR, drop-new и т.п.) собраны в
[словаре терминов](#словарь-терминов) в конце документа.

## Сборка и запуск

Требования: CMake ≥ 3.16 и компилятор C++17 (проверено MSVC 2022+; GCC/Clang -
стандартный C++17 без расширений). Внешних зависимостей нет, doctest в `third_party/`.

```sh
cmake -S . -B build
cmake --build build
```

Демонстрация (10 задач, сквозной сценарий, лог в консоль):

```sh
build/app/Debug/tracker_app.exe      # Windows (Visual Studio, multi-config)
./build/app/tracker_app              # Linux/macOS (Makefiles/Ninja)
```

Unit-тесты:

```sh
cd build && ctest --output-on-failure -C Debug
# либо напрямую: build/tests/Debug/unit_tests.exe
```

Сборка с заглушкой МК - те же исходники, другая реализация HAL
подставляется линковкой:

```sh
cmake -S . -B build-mcu -DTRACKER_PLATFORM=mcu_stub
cmake --build build-mcu
```

## Модель параллелизма

Кооперативный планировщик + изолированная ISR-граница:

- Каждый модуль это задача `core::ITask` с неблокирующим `tick(now_ms)`.
  Планировщик (`core::Scheduler`) обходит статическую таблицу задач
  round-robin на **общем стеке**. Потоки не используются сознательно,
  10 потоков по 1–4 КБ стека съели бы заметную часть 128 КБ и сделали бы
  потребление памяти недетерминированным.
- Асинхронные источники (на железе это прерывания UART/CAN) отделены от
  кооперативного домена lock-free SPSC-кольцами (очередь ровно с одним
  пишущим и одним читающим, без блокировок) - производитель пишет из
  ISR-контекста, потребитель читает из задачи. **Атомики существуют только на
  этой границе**, внутри кооперативного домена всё однопоточно, и
  межмодульные вызовы (uplink забирает записи архива) не требуют
  синхронизации.
- Время задачам подаёт внешний цикл через `run_once(now_ms)`. Модули не
  читают часы платформы, поэтому тестируются детерминированно (демо
  прогоняет 30 «секунд» мгновенно на симулированном времени). Переполнение
  uint32-миллисекунд (~49,7 суток аптайма) обработано знаковой разностью и
  покрыто тестами.

## Схема модулей и потоков данных

```
 [GPS]p ──► GpsTopic ──┬──► [Геозоны] ──► ZoneEventTopic ──┬──► [Архив] ──► [Uplink] ──► ILinkTransport («сервер»)
   ISR-граница         └──► [Скрипты]s                     └──► лог демо      ▲
                                              записи (IRecordSource) ─────────┘
 [АЦП]p ──► [Датчики]

 [CAN]s    [Питание]s    [Конфигурация]s    [Watchdog] ◄── пробы прогресса (gps, sensors, can)

 p — периферия за внедряемым интерфейсом (в тестах и демо — заглушки)
 s — модуль-заглушка (см. «Статус модулей»)
```

Сквозной сценарий - координата → вход/выход геозоны с
гистерезисом → нумерованная запись архива → аплинк. Проверен
интеграционным тестом (`tests/integration/test_end_to_end.cpp`) и виден в
логе демо. Порядок регистрации задач повторяет поток данных, поэтому событие
проходит цепочку за один круг планировщика.

## Примитивы ядра

| Примитив | Назначение | Политика переполнения                                       |
|---|---|-------------------------------------------------------------|
| `RingBufferSpsc<T,N>` | канал ISR → задача, lock-free (acquire/release) | drop-new + счётчик потерь - как аппаратный FIFO при overrun |
| `Topic<T,S,N>` | один-ко-многим: каждому подписчику своё SPSC-кольцо | медленный подписчик теряет только свои сообщения            |
| `RingBufferOverwrite<T,N>` | хранилище внутри одной задачи (архив) | drop-old - вытеснение старейших                             |

Почему в транспорте drop-new, а не drop-old: вытеснение со стороны
производителя требовало бы двигать индекс потребителя - это ломает lock-free
SPSC-инвариант. Переполнение в штатном режиме недостижимо по построению
(ёмкости выбираются из известных частот источников), а срабатывание политики - признак
больной задачи, который виден диагностике через счётчики потерь.

`Subscription` - move-only хэндл, два потребителя одного кольца исключены на
уровне типов. Подписка и регистрация задач - только на этапе инициализации.

## Бюджет ОЗУ

Способ удержания в 128 КБ: **кучи нет вообще**, все ёмкости (очереди, зоны,
пробы, батчи) — шаблонные константы, поэтому след системы полностью виден
компилятору. `app/ram_budget.hpp` — единая точка учёта: constexpr-сумма
`sizeof` всех системных объектов и `static_assert`, обрывающий сборку при
превышении бюджета. Потребление не зависит от объёма входных данных: при
любом потоке сообщений память ограничена этими же константами.

Текущий след (MSVC x64; на 32-битном МК указатели вдвое короче — след ниже):

| Объект | Байт |
|---|---|
| GpsTopic (4 подписчика × 8 слотов) | 584 |
| ZoneEventTopic (4 × 8) | 584 |
| Scheduler<10> | 248 |
| GpsSimTask | 64 |
| GeofenceTask (8 зон) | 240 |
| ArchiveTask (кольцо на 32 записи) | 832 |
| UplinkTask (батч 8 записей) | 256 |
| SensorPollTask | 32 |
| WatchdogTask (16 проб) | 816 |
| **Итого** | **3656 из 98304 доступных** |

Резервы: 16 КБ стек (один общий стек цикла и ISR, главный потребитель -
printf-лог и вложенные прерывания) и 16 КБ статика рантайма (reent/stdio
newlib, будущие DMA-буферы UART/модема). Обе величины - консервативные
оценки сверху, при портировании на конкретный МК заменяются измеренными:
статика - из linker map (`-Wl,--print-memory-usage`), стек - закраской с
контролем high-water mark. Запас ~27× позволяет наращивать ёмкости в
`msg/topics.hpp` и модулях, пока `static_assert` молчит.

## Допущения и компромиссы

- **Геометрия зон** - планарное приближение в единицах 1e-7° без учёта
  сжатия долготы. Замена метрики локализована в одной функции движка.
- **Полное зависание `tick()`** софтовый watchdog не ловит - кооперативный
  цикл замирает целиком. Это принципиальная граница модели, на железе её
  закрывает аппаратный watchdog, сброс которого разрешается только при
  `system_healthy()`.
- **ISR-путь в демо не поднимает второй поток**: корректность SPSC-кольца под
  реальной конкуренцией доказана стресс-тестом на std::thread (200 тыс.
  сообщений), а в однопоточном демо имитация прерываний потоком не добавила бы
  гарантий.
- **Топики single-publisher** - по текущей схеме потоков у каждого топика один
  производитель. MPSC не требуется и не реализован.
- **printf только в точке входа и демо-обвязке** - модули не логируют сами и
  не зависят от stdio.
- **Резервы стека/статики не измерены** - консервативные оценки, метод замены
  описан в разделе бюджета.
- Сообщения каналов обязаны быть trivially copyable (побайтово) - закреплено
  `static_assert` в `ram_budget.hpp`.

История коммитов сознательно отражает эволюцию решений, в частности, пересмотр
контракта `ITask::tick()` → `tick(now_ms)` (тест сенсоров по реальным
часам оказался медленным и недетерминированным) и добавление гистерезиса
геозон (джиттер-тест вскрыл спам событий у границы зоны).

## Портирование на целевой МК

1. Реализовать `platform_impl` для цели: `now_ms()` от таймера (сейчас есть
   `desktop` на std::chrono и `mcu_stub`-заглушка. Выбор - опцией CMake
   `TRACKER_PLATFORM`, исходники модулей и ядра не меняются).
2. Заменить `GpsSimTask` на парсер, публикующий из UART - контракт
   `Topic` это допускает, публикует единственный контекст-производитель.
3. Подставить реализации `ILinkTransport` (модем) и `ISampleSource` (АЦП).
4. Точка входа: `for(;;) scheduler.run_once(platform::now_ms());` плюс сброс
   аппаратного watchdog при `watchdog.system_healthy()`.

## Словарь терминов

| Термин | Значение                                                                                                                                                |
|---|---------------------------------------------------------------------------------------------------------------------------------------------------------|
| МК | микроконтроллер - целевой процессор устройства                                                                                                          |
| Куча (heap) | динамическая память (`new`/`malloc`) в проекте не используется вовсе                                                                                    |
| Кооперативный планировщик | задачи не вытесняются, а сами быстро возвращают управление: планировщик по кругу вызывает у каждой короткий шаг `tick()`                                |
| Round-robin | обход задач по кругу, каждой по одному шагу за круг                                                                                                     |
| ISR | interrupt service routine - обработчик аппаратного прерывания. «ISR-граница» - место передачи данных из прерывания в обычный код                        |
| SPSC | single producer, single consumer - очередь ровно с одним пишущим и одним читающим контекстом, именно это позволяет обойтись без блокировок              |
| MPSC | multi producer, single consumer - очередь с несколькими пишущими                                                                                        |
| Lock-free | без блокировок (мьютексов): стороны никогда не ждут друг друга, корректность держится на атомарных операциях                                            |
| Атомики, acquire/release | `std::atomic` и режим упорядочивания памяти: потребитель гарантированно видит данные, записанные до публикации индекса                                  |
| Топик, подписчик | канал «один ко многим»: издатель публикует сообщение, каждый подписчик получает собственную копию                                                       |
| drop-new | при переполнении очереди отбрасывается новое (входящее) сообщение, накопленные сохраняются                                                              |
| drop-old | при переполнении вытесняется самое старое сообщение, новое записывается всегда                                                                          |
| FIFO, overrun | first in, first out - выдача в порядке поступления, overrun - переполнение аппаратного приёмного буфера                                                 |
| HAL | hardware abstraction layer - слой доступа к железу; реализация подменяется на этапе линковки                                                            |
| Watchdog | сторожевой механизм, аппаратный - перезагружает МК, если его периодически не сбрасывать, программный - следит, что счётчики работы модулей продвигаются |
| Heartbeat | периодическое сообщение «я жив» от задачи-заглушки                                                                                                      |
| Гистерезис | разнесённые пороги входа и выхода (выход дальше входа), гасит дребезг событий при колебаниях у границы зоны                                             |
| Батч | пакет из нескольких записей, отправляемых за одну передачу                                                                                              |
| seq | сквозной порядковый номер записи; разрыв в номерах - признак вытеснения записей при переполнении архива                                                 |
| DMA | прямой доступ к памяти, периферия пишет данные в память без участия процессора                                                                          |
| newlib, reent | стандартная библиотека C для встраиваемых систем и её статические структуры состояния                                                                   |
| Linker map, high-water mark | карта памяти от компоновщика; отметка максимального заполнения стека («закраска» - заполнить стек паттерном и посмотреть, докуда он затёрт)             |
| move-only | тип можно передать (переместить), но нельзя скопировать - исключает появление второго владельца                                                         |
| trivially copyable | тип копируется побайтово, без пользовательских конструкторов - безопасен для копирования в кольцевые буферы                                             |
| static_assert | проверка условия на этапе компиляции: при нарушении сборка останавливается с заданным сообщением                                                        |
